> Для локального запуска и разработки см. [README.md](README.md) (раздел «Быстрый старт») и [ИНСТРУКЦИЯ_ЗАПУСКА.md](ИНСТРУКЦИЯ_ЗАПУСКА.md). Этот документ — только про production.

## 1. Архитектура production

```mermaid
flowchart LR
    I[Интернет] -->|80/443| NG[Nginx<br/>reverse-proxy + TLS]
    NG -->|web:5000| WEB[Flask + Gunicorn/Waitress]
    WEB --> PG[(PostgreSQL 16)]
    WEB -. опц. .-> RD[(Redis)]
    RD -. опц. .-> WK[Celery worker]
    WEB -->|/metrics| PR[Prometheus]
    NODE[node-exporter] --> PR
    PR --> GR[Grafana]
    PT[promtail] --> LK[Loki] --> GR
    NG -->|/updates/| UPD[Установщик десктопа]
```

| Компонент | Контейнер / процесс | Назначение |
|-----------|---------------------|------------|
| Reverse-proxy | `nginx` | TLS-терминация, rate-limit `/api/`, раздача `/static/` и `/updates/`, порты 80/443 |
| Веб-приложение | `web` (Flask + Gunicorn на Unix / Waitress на Windows) | Auth, API для десктопа, админ-панель, асинхронный экспорт |
| База данных | `db` (PostgreSQL 16) | Школы, пользователи, оценки (`GradeReport`), задачи |
| Очередь (опц.) | `redis` + `worker` (профиль `async`) | Celery для тяжёлого экспорта/AI; по умолчанию выключено (`USE_CELERY=0`, задачи идут в потоках) |
| Метрики | `prometheus`, `node-exporter`, `grafana` | Сбор и визуализация метрик приложения и хоста |
| Логи | `loki`, `promtail` | Централизованные JSON-логи приложения |
| Трекинг ошибок (опц.) | GlitchTip (отдельный compose) | Sentry-совместимый сбор исключений |

Важная особенность: тяжёлый Playwright-скрапинг выполняется **на компьютере учителя** в десктоп-клиенте, а не на сервере. Сервер — это API + хранилище отчётов + аналитическая панель. Поэтому серверу не нужен Chromium и большие ресурсы под браузеры.

## 2. Требования к серверу

**Минимум (до ~20 школ):**

- 2 vCPU, 4 ГБ RAM, 40 ГБ SSD
- ОС: Ubuntu 22.04/24.04 LTS (рекомендуется) или другой Linux с Docker
- Docker Engine 24+ и Docker Compose v2 (`docker compose`, не `docker-compose`)
- Открытые порты: `80`, `443` (наружу), `9090`/`3000` — только для администратора (закрыть firewall'ом или пробрасывать через SSH-туннель)

**Рекомендуется (production):**

- 4 vCPU, 8 ГБ RAM, 80 ГБ SSD
- Домен с A-записью на сервер (например, `mektep-analyzer.kz`)
- Резервное копирование тома PostgreSQL (см. [раздел 10](#10-резервное-копирование-и-восстановление))

Без Docker (bare-metal) — см. [раздел 12](#12-альтернатива-развёртывание-без-docker-bare-metal). Тогда нужны Python 3.11+ и PostgreSQL 16 на хосте.

## 3. Подготовка сервера

На чистом Ubuntu:

```bash
# Обновление системы
sudo apt update && sudo apt upgrade -y

# Установка Docker Engine + Compose v2
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER
newgrp docker   # или перелогиньтесь

# Проверка
docker --version
docker compose version

# Firewall: открываем только web-порты
sudo ufw allow OpenSSH
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable
```

> Порты Prometheus (9090) и Grafana (3000) **не** открывайте наружу. Доступ к ним — через SSH-туннель: `ssh -L 3000:localhost:3000 -L 9090:localhost:9090 user@server`.

## 4. Получение кода и конфигурация `.env`

```bash
git clone <repo-url> /opt/mektep
cd /opt/mektep
cp env.example .env
```

Сгенерируйте секреты:

```bash
# SECRET_KEY (32 байта hex)
python3 -c "import secrets; print('SECRET_KEY=' + secrets.token_hex(32))"

# PASSWORD_ENC_KEY (ключ Fernet для шифрования отображаемых паролей учителей)
python3 -c "from cryptography.fernet import Fernet; print('PASSWORD_ENC_KEY=' + Fernet.generate_key().decode())"

# Пароль PostgreSQL
python3 -c "import secrets; print('POSTGRES_PASSWORD=' + secrets.token_hex(16))"
```

Отредактируйте [.env](env.example). Минимально обязательное для production:

```ini
FLASK_ENV=production

# Секреты (НЕ коммитить, НЕ оставлять дефолтными!)
SECRET_KEY=<сгенерированный выше>
PASSWORD_ENC_KEY=<сгенерированный выше>
POSTGRES_PASSWORD=<сгенерированный выше>

# В Docker DATABASE_URL формируется автоматически из POSTGRES_PASSWORD,
# указывать его в .env для compose не обязательно.

# Первый суперадмин (создаётся при первом старте, если суперадмина ещё нет)
BOOTSTRAP_SUPERADMIN_USER=admin
BOOTSTRAP_SUPERADMIN_PASS=<надёжный-временный-пароль>

# Загрузки/хранилище в контейнере
UPLOAD_ROOT=/app/data/uploads

# Лимиты задач (на сервере тяжёлого скрапинга нет, но экспорт/потоки используют)
MAX_CONCURRENT_JOBS=5
JOB_TIMEOUT_SECONDS=1800

# Наблюдаемость
LOG_LEVEL=INFO
LOG_JSON=1
GRAFANA_ADMIN_PASSWORD=<надёжный-пароль-grafana>

# AI (опционально; индивидуальный ключ школы задаётся в админке)
DASHSCOPE_API_KEY=

# Кнопка скачивания десктопа (по умолчанию ведёт на установщик в /updates/)
# DESKTOP_DOWNLOAD_URL=https://<ваш-домен>/updates/MektepDesktopSetup-<ver>.exe
```

> **Критично:** в production `ProductionConfig` требует непустой `SECRET_KEY` и `DATABASE_URL` ([webapp/config.py](webapp/config.py)). Если оставить `SECRET_KEY` дефолтным — сессии скомпрометированы. После первого входа **обязательно** смените пароль суперадмина и при желании удалите `BOOTSTRAP_SUPERADMIN_PASS` из `.env`.

Полный справочник переменных окружения — в разделе «Конфигурация (веб)» [README.md](README.md) и в [env.example](env.example).

## 5. Развёртывание через Docker Compose (рекомендуется)

Используйте **корневой** [docker-compose.yml](docker-compose.yml) и запускайте из корня репозитория — он полный (правильный контекст сборки, монтирование `./updates` в Nginx, корректные пути к `deploy/grafana`, `deploy/loki-config.yml`, `deploy/promtail-config.yml`, профиль `async` для Celery).

> В репозитории есть второй файл [deploy/docker-compose.yml](deploy/docker-compose.yml) — это урезанный дубликат для запуска изнутри `deploy/`. Для production используйте корневой файл, как показано ниже.

### 5.1. Базовый стек (Nginx + web + PostgreSQL + мониторинг)

```bash
cd /opt/mektep

# Подготовьте каталоги, которые монтируются в контейнеры
mkdir -p data/uploads instance updates

# Сборка и запуск
docker compose up -d --build

# Статус и логи
docker compose ps
docker compose logs -f web
```

Что поднимется (корневой compose):

| Сервис | Порт (host) | Назначение |
|--------|-------------|------------|
| `nginx` | 80, 443 | Reverse-proxy на `web:5000`, rate-limit `/api/`, раздача `/updates/` |
| `web` | — (внутри сети) | Flask-приложение, `USE_CELERY=0` |
| `db` | — (не проброшен) | PostgreSQL 16, healthcheck `pg_isready` |
| `node-exporter` | — | Метрики хоста |
| `prometheus` | 9090 | Сбор метрик |
| `grafana` | 3000 | Дашборды |
| `loki` / `promtail` | — | Централизованные логи |

Данные:
- PostgreSQL → именованный том `postgres-data`
- Grafana → `grafana-data`, Loki → `loki-data`
- Отчёты/загрузки → хост-каталог `./data` (`/app/data` в контейнере)
- SQLite/instance → `./instance` (для PostgreSQL не используется, но монтируется)

### 5.2. Опционально: асинхронный экспорт через Celery + Redis

По умолчанию фоновые задачи (экспорт Excel) идут в потоках внутри `web` (`USE_CELERY=0`) — этого достаточно для большинства школ. Если нужен отдельный worker (Celery), поднимите профиль `async`:

```bash
docker compose --profile async up -d --build
```

Это добавит сервисы `redis` и `worker` (`celery -A webapp.celery_app worker`). При этом `web` остаётся с `USE_CELERY=0` — при необходимости переключите его на Celery, задав `USE_CELERY=1` и `REDIS_URL=redis://redis:6379/0` в окружении `web` (compose-файл/`.env`).

### 5.3. Проверка работоспособности

```bash
# Liveness/readiness через Nginx
curl -fsS http://localhost/health/live
curl -fsS http://localhost/health/ready     # 200 = БД доступна

# Сводная статистика
curl -fsS http://localhost/health/stats
```

## 6. HTTPS / TLS

Боевой стенд обязан работать по HTTPS. Два рабочих варианта.

### Вариант A. Терминация TLS на Nginx внутри Compose

1. Получите сертификаты (например, Let's Encrypt через `certbot` на хосте) и положите их в `nginx/ssl/`:

   ```bash
   sudo certbot certonly --standalone -d mektep-analyzer.kz
   mkdir -p nginx/ssl
   sudo cp /etc/letsencrypt/live/mektep-analyzer.kz/fullchain.pem nginx/ssl/
   sudo cp /etc/letsencrypt/live/mektep-analyzer.kz/privkey.pem   nginx/ssl/
   ```

2. В корневом [docker-compose.yml](docker-compose.yml) у сервиса `nginx` раскомментируйте монтирование сертификатов:

   ```yaml
   volumes:
     - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
     - ./updates:/var/www/mektep/updates:ro
     - ./nginx/ssl:/etc/nginx/ssl:ro
   ```

3. В [nginx/nginx.conf](nginx/nginx.conf) раскомментируйте `server { listen 443 ssl http2; ... }`, пропишите `server_name`, скопируйте в него `location`-блоки из HTTP-сервера и добавьте редирект 80→443. HSTS-заголовок уже заготовлен.

4. Перезапустите Nginx: `docker compose up -d nginx`. Сертификаты Let's Encrypt продлевайте по cron (`certbot renew`) с последующим `docker compose restart nginx`.

> Каталог `nginx/ssl/` исключён из git (см. `.gitignore`) — сертификаты никогда не попадут в репозиторий.

### Вариант B. Внешний TLS (Cloudflare / внешний LB)

Поставьте Cloudflare (или облачный балансировщик) перед сервером и терминируйте TLS там, проксируя на порт 80. В этом случае:

- Для путей `/health/*` поставьте в Cloudflare `Cache Level: Bypass` (иначе мониторинг увидит закэшированный статус).
- Nginx уже передаёт `X-Forwarded-Proto`/`X-Forwarded-For`.

## 7. Инициализация БД и первый вход

Схема БД поднимается автоматически при старте `web`: фабрика `create_app()` выполняет `db.create_all()` и «лёгкие» runtime-ALTER недостающих колонок ([webapp/__init__.py](webapp/__init__.py)). Отдельные команды миграций обычно не требуются.

Для существующих инсталляций (обновление со старой схемы) можно выполнить идемпотентные миграции явно:

```bash
docker compose exec web python -m scripts.db.add_ai_api_key_column
docker compose exec web python -m scripts.db.add_progress_columns
docker compose exec web python -m scripts.db.add_final_report_data
docker compose exec web python -m scripts.db.add_academic_year_column
```

Первый суперадмин создаётся автоматически из `BOOTSTRAP_SUPERADMIN_USER/PASS` при первом старте. Создать/добавить вручную:

```bash
docker compose exec web flask --app entrypoints.app create-superadmin <user> <pass>
```

Дальнейшие шаги в браузере (`https://<домен>/`):

1. Войдите суперадмином и **сразу смените пароль**.
2. Создайте школу (раздел «Суперадмин»), активируйте её, при необходимости задайте AI-ключ школы.
3. Создайте школьного админа для каждой школы.
4. Под школьным админом добавьте учителей, классы и предметы.
5. Учителя входят **только** через десктоп-клиент; их учётные данные используются для входа в `mektep.edu.kz` через десктоп.

## 8. Публикация десктоп-клиента

Учителя скачивают установщик с сервера (кнопка «Скачать приложение» на лендинге → файл в `/updates/`), а клиент автообновляется через манифест `latest.json`.

1. Соберите установщик на **Windows-машине** (Inno Setup 6 + PyInstaller):

   ```bash
   cd mektep-desktop
   python build.py        # → dist/MektepDesktopSetup-<ver>.exe + dist/latest.json
   ```

2. Залейте оба файла на сервер в каталог `updates/` (он монтируется в Nginx как `/var/www/mektep/updates/`):

   ```bash
   scp dist/MektepDesktopSetup-<ver>.exe dist/latest.json deploy@server:/opt/mektep/updates/
   ```

3. Проверьте раздачу:

   ```bash
   curl -fsS https://<домен>/updates/latest.json
   ```

`latest.json` отдаётся с `no-cache`, поэтому клиенты сразу видят новую версию. Регламент релиза и формат манифеста — [mektep-desktop/UPDATES.md](mektep-desktop/UPDATES.md) и [mektep-desktop/RELEASE.md](mektep-desktop/RELEASE.md). При выпуске версии обновите `APP_VERSION` в [mektep-desktop/version.py](mektep-desktop/version.py) и при необходимости `DESKTOP_VERSION`/`MIN_DESKTOP_VERSION` в [webapp/constants.py](webapp/constants.py) (минимальная версия блокирует старые клиенты через `/api`).

## 9. Мониторинг и логи

### Health endpoints
- `GET /health/live` — liveness (для балансировщиков/UptimeRobot).
- `GET /health/ready` — readiness, проверяет БД (200/503).
- `GET /health/stats` — счётчики задач, школ, пользователей.
- `GET /metrics` — Prometheus-метрики.

Настройте внешний аптайм-чек (UptimeRobot) на `https://<домен>/health/live`, интервал 5 мин.

### Prometheus + Grafana
- Prometheus: `http://localhost:9090` (через SSH-туннель), `/targets` — все должны быть `UP`.
- Grafana: `http://localhost:3000`, логин `admin`, пароль из `GRAFANA_ADMIN_PASSWORD`. Источники Prometheus и Loki подключаются автоматически ([deploy/grafana/provisioning](deploy/grafana/provisioning/datasources/datasources.yml)).
- Импортируйте дашборды: `1860` (Node Exporter Full), `9628`/`9964` (PostgreSQL), кастомный на базе `flask_http_request_*`.

### Логи (Loki / Grafana Explore)
Приложение пишет структурированные JSON-логи в stdout (`request_id`, `duration_ms`, `user_id`, `school_id`). Сбор: promtail → Loki → Grafana. Быстрый просмотр:

```bash
docker compose logs -f web        # приложение
docker compose logs -f nginx      # access/error Nginx
```

### Трекинг ошибок (опционально, GlitchTip)

```bash
docker compose -f deploy/glitchtip-compose.yml up -d
```

Создайте проект, скопируйте DSN в `.env` (`SENTRY_DSN=...`) и перезапустите `web`: `docker compose up -d web`.

Подробные PromQL/LogQL-примеры — в разделе «Мониторинг» [README.md](README.md).

## 10. Резервное копирование и восстановление

Критичные данные: **БД PostgreSQL** (том `postgres-data`) и **каталог загрузок** `./data` (готовые отчёты).

### Бэкап БД (ежедневно по cron)

```bash
# Дамп PostgreSQL
docker compose exec -T db pg_dump -U mektep mektep_db | gzip > backup-db-$(date +%F).sql.gz

# Архив отчётов
tar czf backup-data-$(date +%F).tar.gz data/
```

Пример crontab (3:00 ночи, хранение 14 дней):

```cron
0 3 * * * cd /opt/mektep && docker compose exec -T db pg_dump -U mektep mektep_db | gzip > /backups/db-$(date +\%F).sql.gz && find /backups -name 'db-*.sql.gz' -mtime +14 -delete
```

### Восстановление БД

```bash
gunzip -c backup-db-2026-06-30.sql.gz | docker compose exec -T db psql -U mektep -d mektep_db
```

Если БД была утеряна, но файлы отчётов на диске сохранились, можно восстановить записи `ReportFile`:

```bash
docker compose exec web python -m scripts.db.recover_reports
```

## 11. Обновление приложения

```bash
cd /opt/mektep
docker compose exec -T db pg_dump -U mektep mektep_db | gzip > backup-before-upgrade.sql.gz  # бэкап!

git pull
docker compose up -d --build        # пересборка изменённых образов

docker compose logs -f web          # проверьте старт без ошибок
curl -fsS http://localhost/health/ready
```

Схема обновляется автоматически (`db.create_all()` + runtime-ALTER). При мажорных изменениях схемы примените соответствующие скрипты из `scripts/db/` (см. [раздел 7](#7-инициализация-бд-и-первый-вход)).

Откат: `git checkout <предыдущий-тег> && docker compose up -d --build`, при необходимости восстановите БД из бэкапа.

## 12. Альтернатива: развёртывание без Docker (bare-metal)

Если Docker недоступен. Нужны Python 3.11+ и PostgreSQL 16 на хосте.

```bash
# 1. Окружение
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
# Chromium нужен только если на этом же хосте запускается CLI-скрапер:
# python -m playwright install chromium

# 2. PostgreSQL: создайте БД и пользователя
sudo -u postgres psql -c "CREATE USER mektep WITH PASSWORD 'СЕКРЕТ';"
sudo -u postgres psql -c "CREATE DATABASE mektep_db OWNER mektep;"

# 3. .env
cp env.example .env
# FLASK_ENV=production
# SECRET_KEY=..., PASSWORD_ENC_KEY=...
# DATABASE_URL=postgresql://mektep:СЕКРЕТ@localhost:5432/mektep_db

# 4. Запуск production-сервера (Gunicorn на Unix, Waitress на Windows)
python run_production.py
```

Либо явно:

```bash
# Unix
gunicorn -c entrypoints/gunicorn_config.py entrypoints.wsgi:app

# Windows
python -c "from waitress import serve; from entrypoints.wsgi import app; serve(app, host='0.0.0.0', port=5000, threads=4)"
```

Параметры сервера переопределяются переменными `GUNICORN_*` / `WAITRESS_*` ([entrypoints/gunicorn_config.py](entrypoints/gunicorn_config.py)).

### systemd-юнит (Linux)

`/etc/systemd/system/mektep.service`:

```ini
[Unit]
Description=Mektep Scraper web
After=network.target postgresql.service

[Service]
User=mektep
WorkingDirectory=/opt/mektep
EnvironmentFile=/opt/mektep/.env
```

## 14. Устранение неполадок

| Симптом | Причина / решение |
|---------|-------------------|
| `web` падает при старте, лог `SQLALCHEMY_DATABASE_URI ... None` | В production не задан `DATABASE_URL`. В compose он формируется из `POSTGRES_PASSWORD` — проверьте `.env`. |
| `502 Bad Gateway` от Nginx | `web` ещё не поднялся или упал. `docker compose logs web`, дождитесь healthcheck `db`. |
| `/health/ready` отдаёт 503 | Нет связи с БД. Проверьте контейнер `db` (`docker compose ps`, `pg_isready`). |
| `/static/` отдаёт 404 через Nginx | Nginx не имеет доступа к статике: либо смонтируйте `./webapp/static` в контейнер `nginx` как `/app/webapp/static/`, либо уберите `location /static/` (Flask отдаёт статику сам). CSS подключается с CDN, поэтому критичного влияния обычно нет. |
| `429 Too Many Requests` на `/api/` | Сработал rate-limit Nginx. Это ожидаемо при всплесках; при необходимости поднимите `rate`/`burst` в [nginx/nginx.conf](nginx/nginx.conf). |
| Кнопка «Скачать приложение» ведёт на 404 | В `updates/` нет `MektepDesktopSetup-<DESKTOP_VERSION>.exe`. Залейте установщик (см. [раздел 8](#8-публикация-десктоп-клиента)) или задайте `DESKTOP_DOWNLOAD_URL`. |
| Десктоп-клиенты не видят обновление | Проверьте `https://<домен>/updates/latest.json` и поле `version`; убедитесь, что версия выше `APP_VERSION` клиентов. |
| Учитель не может войти на сайт | Так и задумано: учителя входят только через десктоп. На сайте работают суперадмин и школьные админы. |
| Grafana: «no data» | Проверьте `http://localhost:9090/targets` — все таргеты должны быть `UP`; источники данных в Grafana подключены автоматически. |
| AI-генерация не работает | Не задан AI-ключ школы (суперадмин → школа) или глобальный `DASHSCOPE_API_KEY`. |

Логи для диагностики:
- Приложение: `docker compose logs -f web` (JSON) или Grafana → Explore → Loki.
- Nginx: `docker compose logs -f nginx`.
- БД: `docker compose logs -f db`.
