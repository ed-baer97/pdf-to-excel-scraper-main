## 3. Постановка задачи (черновик для наполнения)

### 3.1 Основание для разработки

В образовательной организации регулярно формируются отчеты по учебному процессу на основе данных электронного журнала. При ручной подготовке документов значительная часть времени уходит на однотипные действия: сбор сведений, перенос в таблицы и оформление итоговых файлов. Это увеличивает нагрузку на педагогов и администраторов, а также повышает риск технических ошибок.

Разрабатываемая авторская программа предназначена для автоматизации указанных операций: получения данных с портала mektep.edu.kz и формирования отчетных документов в форматах Excel и Word по заранее подготовленным шаблонам.

### 3.2 Цель и назначение программы

Цель разработки — сократить трудозатраты при подготовке школьной отчетности и обеспечить единообразное формирование документов.

Назначение программы:
- автоматизированный сбор данных с образовательного портала;
- преобразование полученных данных в структурированные отчеты;
- предоставление удобного интерфейса работы для сотрудников школы.

### 3.3 Функциональные требования

Программа должна обеспечивать:
- авторизацию пользователей и разграничение доступа по ролям (администрирование и работа учителя);
- запуск процедуры сбора данных с портала mektep.edu.kz;
- формирование отчетов в форматах Excel и Word на основе шаблонов;
- отображение статуса выполнения задач и доступ к готовым файлам;
- хранение и управление сформированными отчетами в системе.

### 3.4 Нефункциональные требования

К системе предъявляются следующие нефункциональные требования:
- корректная работа на типовой школьной технике под Windows;
- понятный интерфейс, не требующий длительного обучения;
- устойчивость к временным сбоям сети и внешнего сервиса;
- возможность локального запуска для разработки и серверного развертывания для школы;
- соблюдение базовых требований информационной безопасности при работе с учетными данными.

### 3.5 Пользователи и роли

В рамках проекта выделяются следующие категории пользователей:
- администратор (настройка учетных записей, контроль работы системы, доступ к отчетам);
- учитель (запуск сбора данных и получение собственных отчетов).

Для каждой роли предоставляется только необходимый набор функций, что уменьшает вероятность ошибок при работе.

### 3.6 Входные и выходные данные

Входные данные:
- учетные данные пользователя для работы в системе;
- параметры формирования отчета (период, язык, шаблон и другие доступные настройки).

Выходные данные:
- сформированные отчетные файлы Excel и Word;
- сведения о выполненных задачах и статусе обработки.

### 3.7 Ограничения и условия применения

Ограничения проекта:
- работоспособность зависит от доступности портала mektep.edu.kz и стабильности интернет-соединения;
- корректность итогового документа зависит от актуальности используемых шаблонов;
- при изменениях структуры внешнего портала может потребоваться адаптация модуля сбора данных.

Условия применения:
- установленная среда Python и необходимые зависимости проекта;
- корректно настроенное окружение (база данных, при необходимости фоновые сервисы);
- соблюдение регламента использования учетных данных образовательной организации.

### 3.8 Критерии успешности

Результат разработки считается достигнутым, если:
- пользователь может запустить сбор данных и получить готовый отчет без ручного переноса информации;
- сформированные файлы соответствуют заданным шаблонам;
- основные сценарии работы выполняются стабильно в учебных условиях.

## 4. Выбор средств разработки (черновик для наполнения)

### 4.1 Критерии выбора технологического стека

При выборе инструментов для реализации программы учитывались следующие критерии:
- доступность технологий и наличие подробной документации;
- скорость разработки и возможность быстрого внесения изменений;
- совместимость с операционной системой Windows, используемой в школах;
- надежность работы с веб-интерфейсами и автоматизацией браузера;
- возможность масштабирования решения от локального запуска до серверного развертывания.

С учетом указанных критериев был выбран стек на базе языка Python и его экосистемы.

### 4.2 Язык программирования Python

В качестве основного языка разработки выбран Python. Он обеспечивает:
- высокую скорость создания прикладных решений;
- читаемый и поддерживаемый код;
- широкий выбор библиотек для веб-разработки, автоматизации и обработки документов;
- удобство сопровождения проекта в образовательной среде.

Python также позволяет объединить в одном проекте серверную часть, модуль автоматизированного сбора данных и инструменты формирования отчетов.

### 4.3 Веб-фреймворк Flask

Для реализации веб-платформы использован Flask. Выбор обусловлен следующими преимуществами:
- минималистичная архитектура и гибкость при проектировании приложения;
- удобная интеграция с шаблонами, ORM и системой авторизации;
- низкий порог входа для поддержки и доработки;
- возможность поэтапного развития проекта без избыточной сложности.

Flask используется для реализации пользовательского интерфейса, маршрутов, работы с ролями пользователей и управления задачами формирования отчетов.

### 4.4 Автоматизация браузера: Playwright

Для получения данных с портала mektep.edu.kz выбран Playwright, так как портал является динамическим веб-приложением и требует полноценного управления браузером.

Преимущества Playwright в рамках проекта:
- стабильная работа с современными JavaScript-страницами;
- поддержка сценариев авторизации и последовательной навигации;
- возможность запуска в headless-режиме и в режиме визуальной отладки;
- инструменты ожидания элементов, снижающие число ошибок при загрузке страниц.

Таким образом, Playwright обеспечивает корректный сбор данных в условиях реального веб-интерфейса.

### 4.5 Система управления базами данных

Для хранения учетных записей, задач и метаданных отчетов предусмотрена реляционная СУБД.

Режимы использования:
- SQLite - для локальной разработки и тестирования;
- PostgreSQL - для промышленного использования в школьной инфраструктуре.

Данный подход позволяет сочетать простоту запуска на этапе разработки и надежность хранения данных в рабочей среде.

### 4.6 Библиотеки формирования отчетов

Формирование итоговых документов реализуется с применением библиотек Python для работы с офисными форматами:
- для Excel-отчетов используются инструменты чтения и заполнения шаблонных таблиц;
- для Word-отчетов применяются библиотеки генерации документов на основе шаблонов.

Использование шаблонного подхода обеспечивает единый стиль оформления документов и уменьшает количество ручных правок после генерации.

### 4.7 Средства разработки десктоп-клиента

Для пользовательских сценариев, где удобнее локальное приложение, используется десктоп-клиент на базе PyQt6.

Выбор обусловлен следующими причинами:
- нативный графический интерфейс для Windows;
- удобная интеграция с Python-кодом основного проекта;
- поддержка многопоточности для длительных операций без "зависания" интерфейса;
- возможность сборки исполняемого файла для конечных пользователей.

Наличие десктоп-клиента расширяет варианты внедрения программы в школах с различными организационными требованиями.

### 4.8 Инструменты тестирования и сопровождения

Для обеспечения качества и стабильности проекта применяются:
- pytest - для автоматизированного тестирования отдельных модулей;
- встроенные средства логирования - для диагностики ошибок и контроля выполнения задач;
- система управления версиями Git - для отслеживания изменений и командной работы.

Использование данных инструментов повышает надежность программного комплекса и упрощает дальнейшее развитие функциональности.

### 4.9 Вывод по разделу

Выбранный набор средств разработки соответствует целям проекта и условиям эксплуатации в образовательной организации. Комбинация Python, Flask, Playwright, реляционной СУБД и библиотек генерации документов позволяет реализовать полный цикл: от получения данных из внешнего портала до формирования готовой отчетной документации в удобном для пользователей формате.

## 6. Описание реализации (черновик для наполнения)

### 6.1 Общий подход к реализации

Реализация программного комплекса выполнена как связка независимых, но согласованных модулей: пользовательского интерфейса, подсистемы запуска задач, скрапинга данных и генерации отчетных документов. Ключевой принцип реализации - разделение ответственности между компонентами, благодаря чему каждую часть можно дорабатывать без изменения всей системы.

Пользователь инициирует задачу через веб-интерфейс или десктоп-клиент, после чего обработка выполняется по унифицированному конвейеру: получение данных с портала mektep.edu.kz, нормализация информации, формирование Excel-отчета и подготовка Word-документа.

### 6.2 Реализация веб-приложения

Веб-часть построена на Flask и обеспечивает:
- аутентификацию пользователей и разграничение прав доступа;
- маршруты для запуска задач формирования отчетов;
- отображение статусов выполнения и истории операций;
- предоставление доступа к готовым отчетным файлам.

Организация веб-слоя выполнена по модульному принципу: отдельные обработчики для ролей пользователей, модели для работы с данными, а также сервисные модули для интеграции с процессом скрапинга и генерации документов.

### 6.3 Реализация модуля сбора данных (скрапер)

Сбор информации реализован на Playwright, что позволяет работать с динамическим интерфейсом портала как с полноценным браузерным сценарием.

Основные этапы реализации скрапера:
- запуск браузерной сессии и авторизация в системе;
- переход по необходимым разделам электронного журнала;
- извлечение учебных показателей и служебных данных;
- проверка корректности полученных значений;
- сохранение структурированных результатов для последующей обработки.

Для повышения устойчивости применяются ожидания загрузки элементов, обработка исключений и повторные попытки для операций, чувствительных к сетевым задержкам.

### 6.4 Реализация формирования Excel-отчетов

Генерация Excel реализована на основе шаблонного подхода:
- в систему загружаются предварительно подготовленные формы документов;
- в шаблоны подставляются данные, полученные модулем скрапинга;
- автоматически рассчитываются поля и заполняются табличные блоки;
- итоговый файл сохраняется в каталоге отчетов с уникальным именем.

Такой подход обеспечивает единообразие внешнего вида отчетов и минимизирует ручные правки после генерации.

### 6.5 Реализация формирования Word-отчетов

Word-документы формируются как производный этап после создания Excel-файла или напрямую из подготовленных структур данных. В рамках реализации:
- выбирается шаблон документа, соответствующий типу отчета;
- в текстовые и табличные блоки подставляются расчетные значения;
- формируются итоговые документы для печати и передачи в администрацию.

Шаблонизация Word-форм позволяет поддерживать единый стиль деловой документации и ускоряет обновление форм при изменении требований.

### 6.6 Реализация сценариев работы пользователя

Типовой сценарий работы пользователя реализован следующим образом:
1. вход в систему под учетной записью своей роли;
2. выбор параметров отчета (период, класс, язык, шаблон);
3. запуск задачи обработки;
4. автоматическое выполнение скрапинга и генерации файлов;
5. получение уведомления о результате и скачивание готового документа.

Для администратора дополнительно реализованы функции контроля задач, управления пользователями и мониторинга корректности формирования отчетов.

### 6.7 Обработка ошибок и журналирование

В реализации предусмотрена базовая система диагностики:
- логирование ключевых этапов выполнения;
- фиксация ошибок авторизации, сетевых сбоев и проблем генерации;
- сохранение статусов задач для последующего анализа.

Наличие журналов упрощает сопровождение системы в условиях школы и позволяет быстро выявлять причины неуспешного формирования отчетов.

### 6.8 Особенности реализации развертывания

Программа поддерживает два режима эксплуатации:
- локальный запуск для тестирования и индивидуальной работы;
- серверное развертывание для коллективного использования в образовательной организации.

Для промышленного режима предусмотрены конфигурации запуска через WSGI/Gunicorn, контейнеризацию (Docker) и интеграцию с проксирующим сервером, что обеспечивает стабильную работу многопользовательского веб-доступа.

### 6.9 Вывод по разделу

Практическая реализация подтверждает работоспособность выбранной архитектуры: данные автоматически извлекаются из внешнего портала, преобразуются и преобразуются в формализованные отчеты без ручного переноса. Модульный подход, шаблонизация документов и разделение ролей пользователей обеспечивают удобство эксплуатации, сопровождения и дальнейшего развития программного комплекса.

## 7. Руководство пользователя (черновик для наполнения)

### 7.1 Назначение раздела

Данный раздел предназначен для конечных пользователей программного комплекса Mektep Scraper и описывает порядок установки, запуска и выполнения типовых операций. Руководство ориентировано на две роли: учитель и администратор.

### 7.2 Минимальные требования к рабочему месту

Для корректной работы рекомендуется:
- операционная система Windows 10/11;
- доступ к сети Интернет для работы с порталом mektep.edu.kz;
- установленный Python (для локального запуска из исходного кода);
- современный браузер (для веб-версии);
- доступ к каталогу сохранения отчетов.

### 7.3 Подготовка и первичная настройка

Перед первым запуском необходимо:
1. получить доступ к проекту (исходный код или готовая сборка);
2. установить зависимости проекта согласно инструкции запуска;
3. проверить файл окружения (`.env`) и параметры подключения к базе данных;
4. убедиться, что каталоги для отчетов и временных файлов доступны для записи.

Если используется серверный режим, администратор дополнительно проверяет параметры WSGI/Gunicorn и конфигурацию проксирования.

### 7.4 Запуск веб-версии

Базовый порядок запуска:
1. открыть терминал в корневом каталоге проекта;
2. активировать виртуальное окружение (если используется);
3. выполнить команду запуска приложения (например, через `app.py` или WSGI-точку входа);
4. открыть в браузере локальный или серверный адрес системы;
5. выполнить вход под своей учетной записью.

После авторизации пользователю становится доступен набор функций согласно назначенной роли.

### 7.5 Запуск десктоп-версии

Для сценария локальной работы:
1. запустить десктоп-клиент из подготовленной сборки или из исходного кода;
2. пройти авторизацию в интерфейсе клиента;
3. выбрать параметры отчета и запустить обработку;
4. дождаться уведомления о завершении;
5. открыть каталог с готовыми файлами.

Десктоп-режим удобен при индивидуальной работе и в условиях ограниченного серверного доступа.

### 7.6 Типовой сценарий работы учителя

Учитель выполняет операции в следующей последовательности:
1. вход в систему;
2. выбор типа отчета, периода и дополнительных параметров;
3. запуск задачи формирования;
4. контроль статуса выполнения (ожидание, выполнение, завершено/ошибка);
5. скачивание или открытие сформированного файла;
6. проверка заполненных данных и передача отчета по внутреннему регламенту школы.

Основная задача роли учителя - быстро получить корректный отчет без ручного переноса информации.

### 7.7 Типовой сценарий работы администратора

Администратор, помимо базовых операций, выполняет:
- создание и управление учетными записями пользователей;
- контроль прав доступа и ролей;
- мониторинг очереди задач и состояния сервисов;
- проверку корректности шаблонов отчетов;
- анализ журналов при возникновении ошибок;
- организацию резервного копирования данных и отчетных файлов.

Роль администратора ориентирована на стабильность системы и поддержку пользователей.

### 7.8 Работа с результатами и файлами отчетов

После успешного завершения задачи пользователь получает доступ к отчетам в форматах Excel и/или Word. Рекомендуется:
- сохранять итоговые файлы в согласованной структуре каталогов;
- использовать единые правила именования документов (дата, класс, тип отчета);
- не изменять шаблонную структуру итогового файла перед официальной передачей;
- хранить архив ранее сформированных отчетов для повторной проверки.

### 7.9 Типовые ошибки и способы устранения

На практике возможны следующие ситуации:
- ошибка авторизации на портале mektep.edu.kz;
- недоступность внешнего сервиса или нестабильное интернет-соединение;
- несоответствие шаблона выбранному типу отчета;
- отсутствие прав доступа к каталогу сохранения файлов;
- внутренние ошибки обработки данных.

Рекомендуемый порядок действий:
1. проверить правильность учетных данных и сетевое подключение;
2. повторить запуск задачи;
3. сверить настройки шаблона и параметров отчета;
4. обратиться к администратору для анализа журналов;
5. при необходимости передать техническому специалисту текст ошибки и время инцидента.

### 7.10 Рекомендации по безопасной эксплуатации

Для соблюдения требований информационной безопасности следует:
- не передавать учетные данные третьим лицам;
- использовать только персональные учетные записи;
- завершать рабочую сессию после окончания работы;
- ограничивать доступ к каталогам с отчетами;
- регулярно обновлять программные компоненты и шаблоны.

### 7.11 Вывод по разделу

Руководство пользователя описывает полный практический цикл работы с системой: от запуска приложения до получения готовой отчетной документации. Наличие четких инструкций для учителя и администратора снижает число эксплуатационных ошибок и повышает эффективность внедрения программного комплекса в образовательной организации.
